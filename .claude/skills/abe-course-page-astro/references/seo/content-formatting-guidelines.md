## 1. Paragraph & Sentence Rules

**P1. Maximum 75 words per paragraph — target 50.**
Two to three sentences. On mobile this produces a comfortable 4–6 line block. If a paragraph exceeds 75 words, split it.

**P2. Maximum 20 words per sentence — target 15.**
Short sentences reduce cognitive load. Vary length for rhythm (one short, one medium) but never let a sentence run past 25 words.

**P3. One idea per paragraph.**
If a paragraph covers two distinct points, it's two paragraphs. The reader should be able to summarise each paragraph in a single phrase.

**P4. Front-load the key information.**
The first sentence of every paragraph should carry the main point — the inverted pyramid. Supporting detail follows. Background comes last (or gets cut).

**P5. Start with the benefit or answer, not the setup.**
Write "You'll receive your certificate within 24 hours" — not "Upon successful completion of all required modules, participants will be issued…"

## 2. Heading Rules

**H1. One H1 per page — the page title.**
Every other heading is H2 or lower. Never skip levels (H2 → H4).

**H2. Place a heading every 2–3 paragraphs (roughly every 150–200 words).**
This creates the "layer-cake" scanning pattern. Long stretches of unbroken text without a heading are walls of text, regardless of paragraph length.

**H3. Write headings as complete, descriptive phrases.**
The heading alone — without reading the body — should tell the scanner what the section covers.

| ❌ Weak heading | ✅ Strong heading |
|---|---|
| Overview | What you'll learn in 6 hours |
| Duration | How long does the course take? |
| Requirements | What you need before you start |
| Pricing | Course price and what's included |
| About | Who delivers this training |

**H4. Front-load headings with information-carrying words.**
Users scan the first 2–3 words. "Important information about refunds" → "Refunds: what you need to know."

**H5. Use question-format headings for FAQ and informational sections.**
Questions mirror how tradies actually think about the topic and perform well for voice search and featured snippets.

## 3. List & Bullet Rules

**L1. Maximum 7 items per list.**
Beyond 7, working memory struggles. If you have more, group into sub-lists under category headings.

**L2. Maximum 10 steps in a procedure.**
For longer processes, break into numbered phases (Phase 1: Before you start, Phase 2: During the course, Phase 3: After completion).

**L3. Keep list items parallel in structure.**
All items should start the same way — all verbs, all nouns, or all questions.

| ❌ Mixed structure | ✅ Parallel structure |
|---|---|
| Complete the online modules | Complete the online modules |
| Assessment is required | Pass the final assessment |
| You will receive your card | Receive your White Card |

**L4. Use bullets for unordered information, numbers for sequences.**
If order matters (steps, rankings, priorities), use numbered lists. For everything else, use bullets.

**L5. Each bullet item should be 1–2 lines on mobile.**
If a bullet needs a full paragraph of explanation, it should be a section with a heading — not a bullet point.

## 4. Visual Hierarchy & Emphasis Rules

**V1. Bold only the single most important phrase per paragraph.**
One bold element per paragraph maximum. Reserve bold for the words a scanner needs to grasp the paragraph's point without reading the rest.

**V2. No more than 30% of text on any page should carry visual emphasis.**
This includes bold, coloured text, icons, and callout boxes combined.

**V3. Use callout boxes for content that must not be missed.**
Use for: safety warnings, key eligibility requirements, important deadlines, and certification details. Not for general information.

Callout box types: Info (Steel #4a5568 border, #f0f2f4 bg), Warning (#d97706, #fef8ed), Success (#2d7a3e, #e8f5ec), RTO disclosure (Slate Navy #2B4570, #f0f2f4).

**V4. Pair icons with key data points for instant recognition.**
Use consistently — same icon style throughout. Common: clock → duration, certificate → qualification, dollar → price, map pin → location, users → who it's for.

**V5. Use alternating section backgrounds to create visual breaks.**
White (#ffffff) → Cool Off-White (#f9fafb) → White → Cool Off-White.

## 5. Progressive Disclosure Rules

**PD1. Use accordions for any list of 4+ expandable items.**
FAQ sections, module breakdowns, policy clauses, eligibility tables.

**PD2. Write accordion headers as self-contained answers to scanning.**
"How long does the course take? — 6 hours online" is better than "Duration" alone.

**PD3. Use tabs to organise 3–6 parallel content categories.**
Don't exceed 6 tabs — beyond that, use accordions or separate sections.

**PD4. Default the most-asked question or most important tab to open.**
First accordion panel and first tab should be pre-expanded/selected.

**PD5. Never hide critical conversion information behind a click.**
Price, duration, certification outcome, and primary CTA must always be visible without expanding anything.

## 6. Mobile-Specific Rules

**M1. Body text minimum 18px on mobile — target 18–20px.**

**M2. Touch targets minimum 48×48px with 8px spacing between.**

**M3. Single-column layout on mobile — no side-by-side text blocks.**

**M4. Keep the primary CTA visible within the first screen.**
On a mobile viewport (~667px height), the hero should contain: headline, 1–2 line description, key details, and the enrol button — all without scrolling.

**M5. Use device-specific visibility to simplify mobile layouts.**
Consider hiding secondary info cards, detailed tables, or decorative elements on phone while keeping them on desktop.

## 7. Plain Language Rules

**PL1. Write at Grade 6–8 reading level.**
Use Hemingway Editor (hemingwayapp.com) to check.

**PL2. Use trade-specific terms the audience already knows.**
"White card," "licence," "site induction," "owner-builder permit" — not jargon for tradies. But strip bureaucratic language.

**PL3. Use active voice and second person.**
"You must complete this before starting on-site" — not "Completion of this module is required prior to commencement of on-site activities."

**PL4. Write FAQ questions in the user's voice.**
"How long does the course take?" not "Course Duration Information."

**PL5. Prefer short, common words.**
Buy not Purchase, Help not Assist, About not Approximately, Start not Commence, Get not Obtain, To not In order to, Before not Prior to, Use not Utilise, Then not Subsequently, If not In the event that.

## 8. Section Structure Rules

**S1. Every section gets a clear heading (H2) and should be scannable in isolation.**
A user who jumps to any section via heading should understand it without reading previous sections.

**S2. Follow the ABE fixed anchor pattern for page structure.**
Trust Bar (Steel) → Pre-Hero Meta Zone (#f9fafb) → Hero Content (white) → [Content sections alternating white/off-white] → Final CTA (Steel) → Footer (Cool Off-White) → Legal Footer (Steel).

**S3. Recommended course landing page section order:**
1. Trust bar with credential count + years in business
2. Hero with outcome headline, key details, and primary CTA
3. "Who this course is for"
4. "What you'll learn" — bullet-pointed outcomes (max 7)
5. "How it works" — 3–5 simple steps
6. Course contents — expandable accordion
7. Social proof — testimonials, stats
8. Provider credentials / RTO disclosure
9. Pricing — plain display with clear CTA
10. FAQ — accordion addressing common objections

**S4. Break any section longer than 400 words into sub-sections with H3 headings.**

**S5. End every content section with a micro-CTA or directional cue.**
"Ready to get started? Enrol now" or "Have questions? See our FAQ below."

## 9. Table & Data Rules

**T1. Use tables for comparison data, not for prose.**
Never put paragraph text inside table cells.

**T2. Maximum 4 columns on mobile — prefer 2–3.**

**T3. Left-align text, right-align numbers.**

**T4. Add row striping or light borders for tables with 5+ rows.**
Alternating row backgrounds (#ffffff / #f9fafb).

---

## 10. Image & Media Rules

**IM1. Use informational images only — no decorative stock photos.**

**IM2. Place full-width images between text sections on mobile.**

**IM3. Always include alt text that describes the image's purpose, not its appearance.**
"White Card certificate issued after completing the online course" — not "Image of a card."

**IM4. Compress images for mobile performance.**
Target under 100KB per image.

## 11. Source Links, Micro-CTAs, and Verification Notes

### Source link treatment (SL1–SL3)

**SL1. Style source links as quiet, dotted-underline citations.**
Source links reference the authoritative origin of a claim. They should be present but visually subordinate — the reader's eye should hit the content first, with the source available for verification.

| Element | Value |
|---|---|
| Font size | 13px (one step below body) |
| Colour | Slate Light (#6b6b6b) |
| Underline style | Dotted |
| Underline colour | Cool Border (#e5e7eb) |
| Underline offset | 3px |

CSS pattern:
```css
.source-link a {
  color: var(--slate-light);
  text-decoration: underline;
  text-decoration-style: dotted;
  text-decoration-color: var(--cool-border);
  text-underline-offset: 3px;
}
```

**SL2. Prefix with "Source:" for single references.**
Format: `Source: [Authority Name — Specific Page](URL)`. For legislation: `Legislation: [Act Name Year](URL)`.

**SL3. Do not wrap source links in `<em>` tags.**
Source links are references, not emphasis. Italicising them adds visual noise without meaning.

---

### Micro-CTA directional pattern (MC1–MC4)

**MC1. Use directional arrows (↑↓) on in-page navigation links.**
When a micro-CTA points the reader to another section on the same page, include a small directional arrow indicating whether the target is above or below the current position.

| Direction | Arrow | When to use |
|---|---|---|
| Target is below | ↓ | Pointing to a later section (e.g., "See eligibility requirements ↓") |
| Target is above | ↑ | Pointing to an earlier section (e.g., "Get started for $179 ↑") |

**MC2. Style the arrow as a subtle inline indicator.**
The arrow should be smaller and lighter than the link text — it's a wayfinding hint, not a design feature.

CSS pattern:
```css
.micro-cta .arrow {
  display: inline-block;
  font-size: 11px;
  margin-left: 2px;
  opacity: 0.55;
}
```

HTML pattern:
```html
<p class="micro-cta">
  Ready to enrol? <a href="#enrol">Get started for $179 <span class="arrow">↑</span></a>
  — or check the <a href="#eligibility">eligibility requirements <span class="arrow">↓</span></a> first.
</p>
```

**MC3. Every content section should end with a micro-CTA or directional cue.**
This reinforces rule S5 (Section Structure Rules). Micro-CTAs keep the reader moving through the page rather than bouncing. Preferred patterns:
- Point down to the next logical section
- Point up to the enrolment/pricing section
- Offer a choice: "Ready to enrol? [↑] — or check [next section] first [↓]"

**MC4. Do not use directional arrows on external links or CTA buttons.**
Arrows are for in-page navigation only. External links and primary CTA buttons have their own visual treatment (› chevron for buttons).

### Inline verification notes (VN1–VN4)

**VN1. Place a verification note below every cluster of government-sourced figures.**
Fees, penalties, application costs, and dated requirements sourced from government authorities get a small verification line directly below them.

**VN2. Styling convention:**

| Element | Value |
|---|---|
| Font size | 12px |
| Colour | Ash (#9b9b9b) |
| Margin bottom | var(--sp-4) / 16px |

HTML pattern:
```html
<p style="font-size: 12px; color: var(--ash); margin-bottom: var(--sp-4);">
  Penalty amounts verified against WorkSafe ACT, March 2026
</p>
```

**VN3. Name the specific authority, not the generic level of government.**
"Verified against Access Canberra" not "Verified against ACT Government". "Verified against WorkSafe ACT" not "Verified against the regulator".

**VN4. One note per figure cluster, not per individual number.**
A pricing table with three government fees gets one verification note below the table. A penalty callout with individual and body corporate amounts gets one note below the callout. Do not scatter verification notes between every line.

## 12. Source Citation & Verification Rules

**SC1. Place inline verification notes directly below government-sourced figures.**
Fees, penalties, eligibility requirements, and building class restrictions must carry a verification note. Format: `[What] verified against [Authority] on DD Month YYYY`. Style: 12px, ash colour.

**SC2. Combine verification notes with source links when both reference the same authority.**
Make the authority name the clickable link inside the verification line. This eliminates the redundant separate source-link row. Keep source links separate when they point to a different document (e.g., specific legislation).

**SC3. Use dotted-underline styling for all source citation links.**
Source links use `text-decoration-style: dotted` with Slate Light colour and 3px underline offset. This visually distinguishes them from navigation links.

**SC4. Group citations at the end of the relevant content block.**
Order: body text → source links (legislation) → inline verification note. If combining, the verification note replaces both lines.

**SC5. Use exact dates on all verification timestamps.**
Always DD Month YYYY (e.g., "10 March 2026"), not month-only. Exact dates are a stronger YMYL signal on volatile government data.

**SC6. Cite every government/legislative claim, and end the page with a consolidated Sources list.**
Citation is claim-wide, not figure-only: any government, legislative, or regulatory fact stated on the page — fees, penalties, permit thresholds, eligibility rules, the regulator's identity, what an Act requires — carries a visible inline citation (a source link and/or a verification note naming the authority). In addition, any page making such claims ends with a page-level `Sources` section (heading + list): one row per authority/instrument, each linking to a primary or acceptable-secondary source (the instrument, the regulator's fee schedule/register, or a regulator-published guide on its official domain — see `quality-gates.md` §6.7 source tiers) with a `verified DD Month YYYY` date. See `quality-gates.md` §4 (hard blockers) and §6.7. This is a publish hard blocker — a government or legislative claim with no visible source does not ship.

## Quick Self-Check Before Publishing

Run through these 14 checks on every page before it goes live:

| # | Check |
|---|---|
| 1 | No paragraph exceeds 75 words |
| 2 | A heading appears every 150–200 words |
| 3 | Every heading is descriptive enough to scan without reading the body |
| 4 | No bullet list exceeds 7 items |
| 5 | Bold text appears no more than once per paragraph |
| 6 | Price, duration, and certification are visible without scrolling on mobile |
| 7 | Body text is 18px+ on mobile |
| 8 | All touch targets are 48×48px minimum |
| 9 | Readability score is Grade 8 or below |
| 10 | Every section ends with a directional cue or micro-CTA |
| 11 | Source links use dotted underlines, not bold or italic styling |
| 12 | Government-sourced figures have inline verification notes |
| 13 | Micro-CTAs include directional arrows (↑↓) for in-page links |
| 14 | Every government/legislative claim carries a visible citation, and the page ends with a consolidated Sources list (hard blocker) |

## LearnWorlds Implementation Cheat Sheet

| Technique | LearnWorlds tool | Notes |
|---|---|---|
| Accordions | Native Accordion widget | Set "one open at a time" for mobile |
| Tabs | Native Tabs widget | Max 6 tabs; first tab pre-selected |
| Callout boxes | Text widget padding + background + border | Native settings, no Code widget needed |
| Section background alternation | Section background colour setting | White / #f9fafb alternation |
| Icon + text pairs | Icon widget + Text widget in columns | Or use Code widget for tighter control |
| Device-specific visibility | Section/widget visibility toggle | Hide secondary content on phone |
| Responsive preview | Device Authoring mode | Always check phone view before publishing |
| Reusable sections | Save as Template or Linked Section | Build once, reuse across pages |
| Full-width image breaks | Image widget set to full-width | Compress to <100KB |
| Heading hierarchy | Text widget heading levels | H1 once, H2 for sections, H3 for sub-sections |
